# Інваріанти, dev і prod

Цей розділ потрібен для найважливішого переходу:
від "я зрозумів API" до "я розумію, що саме не можна ламати в ядрі".

Якщо у runtime немає формальних інваріантів, будь-який рефакторинг перетворюється
на шаманство.
У `@reflex/beta` інваріанти прості, але дуже жорсткі.

## Навіщо взагалі потрібні інваріанти

Інваріант - це факт, який має залишатися істинним завжди
або після конкретної операції.

У beta інваріанти відповідають на чотири питання:

1. Коли вузол вважається свіжим.
2. Що має статися після recompute.
3. Чому один refresh-pass не перераховує той самий node двічі.
4. Чому залежність оновлюється раніше за свого споживача.

Якщо хоча б один із цих пунктів порушений, ви побачите один із типових класів багів:

- старе значення після запису
- подвійний recompute того самого `computed`
- phantom dependency після перемикання гілки
- нескінченний цикл або дуже дивний порядок refresh

## Головні інваріанти beta

У `src/core.ts` вони зафіксовані прямо в коді.
Нижче - людська версія.

### 1. Freshness invariant

Вузол свіжий тоді і лише тоді, коли:

```ts
node.v >= max(source.t)
```

Де:

- `node.v` - epoch останньої успішної перевірки або recompute цього вузла
- `source.t` - epoch останньої реальної зміни кожного upstream-джерела

Інтуїція дуже проста:

- `t` каже: "моє значення справді змінювалося ось тоді"
- `v` каже: "мене перевіряли або перераховували ось тоді"

Якщо якась залежність змінилася пізніше, ніж node востаннє перевіряли,
значить node уже не свіжий.

### 2. Recompute correctness

Після успішного recompute має бути істинно:

```ts
node.v = currentEpoch
node.state не містить Dirty
```

Для `computed` це означає:

- нове значення обчислене
- `v` переведений на поточний runtime epoch
- `Invalid/Obsolete` очищені
- якщо значення справді змінилося, то `t = v`

Для `effect` це означає:

- compute-функція виконана
- можливий cleanup збережений
- `v` переведений на поточний epoch
- dirty-state очищений

Важливо: `effect` не зобов'язаний оновлювати `t`, бо він не є значенням,
яке downstream-вузли читатимуть як memo.

### 3. Single-pass guarantee

У межах одного `ensureFresh()` один node не повинен recompute більше одного разу.

Це особливо важливо для:

- diamond-graph
- shared subtree
- довгих ланцюжків, де один вузол може траплятися через різні шляхи

Саме для цього у вузла є `w`, а в контексту є `workEpoch`.

Сенс такий:

- `w === workEpoch` -> вузол уже доданий у стек цього проходу
- `w === -workEpoch` -> вузол уже був витягнутий і оброблений у цьому проході

Це дешевий спосіб отримати dedupe без важких структур.

### 4. Topology guarantee

Залежний вузол не повинен recompute раніше, ніж будуть приведені в актуальний
стан усі його dirty-джерела.

У beta це досягається не окремим topo-sort, а логікою walk:

1. `ensureFresh(node)` кладе цільовий вузол у stack.
2. Перед recompute перевіряє `firstIn`.
3. Якщо знаходиться dirty-source, поточний вузол відкладається.
4. Спочатку в stack відправляється dirty-source.
5. Лише після цього можливий recompute споживача.

Це не "глобальне топологічне сортування графа", а локально коректний
refresh-path під час читання.
Для lazy runtime цього достатньо.

## Dirty-state: важливе спрощення

У beta `Dirty` - не окремий збережений прапор, а похідна маска:

```ts
Dirty = Invalid | Obsolete
```

Це важливо розуміти правильно.

### `Invalid`

Це не доказ застарілості.
Це лише сигнал:

"по цій гілці був write, downstream, можливо, уже невалідний"

### `Obsolete`

А це вже доказ застарілості:

"якийсь source.t > node.v"

### Чому це добре

Такий розклад залишає write-path дешевим:

- запис не зобов'язаний миттєво все перераховувати
- запис не зобов'язаний доводити застарілість кожного вузла
- запис просто штовхає hint вниз по графу

А справжній доказ freshness відбувається пізніше, під час читання.

## Де ці інваріанти живуть у коді

### `src/walkers.ts`

Головне місце для freshness/topology:

- `markInvalid()` поширює `Invalid`
- `needsUpdateFromSourceT()` перевіряє `source.t > node.v`
- `ensureFresh()` гарантує правильний порядок refresh

### `src/engine/compute.ts`

Головне місце для commit-поведінки `computed`:

- оновлює `payload`
- ставить `v`
- чистить dirty-state
- оновлює `t` лише якщо значення справді змінилося

### `src/engine/effect.ts`

Головне місце для commit-поведінки `effect`:

- викликає попередній cleanup
- запускає нову compute-функцію
- записує новий cleanup
- ставить `v`
- чистить dirty-state

### `src/engine/execute.ts`

Спільний виконавець обчислення:

- вмикає `Computing`
- перемикає `activeComputed`
- запускає dependency tracking
- робить cleanup старих залежностей
- знімає `Computing`

## Dev-перевірки: що саме вони ловлять

У beta dev-mode intentionally містить зайві перевірки.
Вони існують не для бізнес-логіки, а для виявлення архітектурних багів під час розробки.

Основні приклади:

- cycle detection через `Computing`
- `console.assert`, що `e.from.t <= node.v`
- `console.assert`, що у fresh-node немає dirty-state
- помилка, якщо хтось намагається виконати node без `compute`

### Чому ці перевірки корисні

Вони ловлять баги в момент появи, а не через десять викликів.

Наприклад:

- забули очистити dirty-state після recompute
- зламали branch cleanup, і в `firstIn` висять старі залежності
- випадково викликали `recompute()` на signal-node
- отримали цикл, який у prod перетворився б на дивну поведінку

## Чому dev-перевірки не повинні жити в prod

Тому що в runtime з пріоритетом на продуктивність є жорстке правило:

у гарячому шляху мають лишатися тільки перевірки, які потрібні для коректності,
а не для діагностики.

Треба розрізняти два класи логіки.

### 1. Коректнісна логіка

Вона має залишитися в prod.
Без неї runtime просто працюватиме неправильно.

Приклади:

- `isDirtyState(...)`
- перевірка `Object.is(...)` на write
- `needsUpdateFromSourceT(...)`
- cleanup старих залежностей після recompute
- dedupe у scheduler

### 2. Діагностична логіка

Вона потрібна лише для розробки.
Її можна і треба вирізати з прод-збірки.

Приклади:

- `console.assert(...)`
- докладні помилки-інваріанти
- додаткові defensive-guards, які мають бути неможливі
  при коректному внутрішньому використанні API

## Як beta вирізає dev-логіку з production build

Через compile-time прапор `__DEV__`.

Ідея така:

```ts
if (__DEV__) {
  console.assert(...)
}
```

На етапі production build Rollup підставляє `__DEV__ = false`.
Після цього minifier бачить мертву гілку й видаляє її.

Це краще, ніж runtime-змінна оточення, тому що:

- менше branch cost у проді
- менше коду в bundle
- менше шансів випадково залишити safety-логіку в hot path

## Що важливо не переплутати

Є тонкий момент.

Не можна робити так:

"Раз перевірка dev-only, значить і вся пов'язана з нею логіка dev-only".

Неправильно.

Наприклад:

- сам `Computing` flag бере участь у коректному протоколі виконання node
- а ось викидання докладної помилки по ньому може бути dev-only

Тобто треба відділяти:

- внутрішню механіку протоколу
- діагностичний шум поверх неї

## Найнебезпечніші помилки під час рефакторингу

### Помилка 1. Чистити не ті прапори

Якщо замість явного зняття `Invalid | Obsolete` використовувати занадто широку маску,
можна випадково знести `Scheduled`, `Disposed` або `Computing`.

Тому безпечніше мислити так:

```ts
clearDirtyState(node)
```

а не через "магічні" cleanup-маски без контексту.

### Помилка 2. Оновлювати `t` при кожному recompute

Це руйнує семантику version-based selective recomputation.

`t` має означати "значення справді змінилося", а не
"вузол черговий раз перерахувався".

### Помилка 3. Вважати `Invalid` повним доказом stale

Ні.
Це лише propagation hint.

Якщо downstream отримав `Invalid`, але всі його залежності в підсумку виявилися
зі старими `t`, node може очиститися без recompute.

### Помилка 4. Видалити branch cleanup заради уявної оптимізації

Тоді з'являться phantom deps.
Ззовні це виглядатиме так:

- змінили `cond`
- гілка вже перемкнулася
- а старе джерело все ще впливає на computed

Це один із найнеприємніших класів багів у signal-runtime.

## Mental check будь-якої зміни

Перед merge корисно поставити собі чотири короткі питання:

1. Чи може node після refresh залишитися dirty?
2. Чи може один `ensureFresh()` перерахувати один node двічі?
3. Чи може залежний вузол оновитися раніше dirty-source?
4. Чи може `firstIn` зберігати залежності, які вже не читалися в останньому compute?

Якщо хоча б на одне питання відповідь "можливо", значить рефакторинг треба
розбирати ще раз.

## Короткий висновок

`@reflex/beta` не тримається на "магії сигналів".
Він тримається на маленькому наборі строгих інваріантів:

- freshness через `t` і `v`
- дешеве invalidation без eager recompute
- рівно один recompute за pass
- коректний локальний topo-order під час refresh
- акуратне зачищення динамічних залежностей

Поки ці речі збережені, ядро можна розвивати.
Щойно вони розмиваються, runtime починає виглядати працюючим лише випадково.
