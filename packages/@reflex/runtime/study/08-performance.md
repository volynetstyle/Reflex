# Продуктивність reflex

Цей розділ пояснює не "хто швидший взагалі", а чому reflex поводиться
швидко в одних сценаріях і слабше в інших.

Це важливіше за голі цифри.
Цифри з часом змінюються, а модель витрат лишається.

## Головна ідея perf-моделі

`reflex` оптимізує не весь життєвий цикл однаково.
Він робить ставку на таку формулу:

- write-path має бути дуже дешевим
- читання має оновлювати лише справді потрібну частину графа
- непрочитані гілки не повинні перераховуватися "про всяк випадок"

Тобто reflex - це lazy selective runtime.

Якщо сказати ще простіше:

```txt
write cheap
invalidate cheap
pay on read
recompute only what was actually demanded
```

## Де reflex зазвичай виграє

### 1. Wide static graph + часткове читання

Сценарій:

- багато memo/computed
- після запису читається лише частина результатів

Чому reflex добра:

- write не тягне весь граф
- downstream отримує лише `Invalid`
- реально перераховуються лише ті вузли, які потім були прочитані

Якщо ви оновили один source, а потім прочитали 10% графа,
reflex часто платить приблизно за ці 10%, а не за весь fan-out.

### 2. Deep chains

Сценарій:

- довгі ланцюжки `A -> B -> C -> D -> ...`
- після зміни джерела читається кінцевий вузол

Чому reflex добра:

- `ensureFresh()` іде рівно потрібним шляхом
- single-pass guarantee не дає зайвих recompute
- проміжні memo не чіпаються заздалегідь

Для lazy runtime це дуже природний сильний кейс.

### 3. Dynamic dependencies

Сценарій:

```ts
computed(() => cond() ? a() : b())
```

або складніші розгалужені залежності.

Чому reflex добра:

- tracking зберігається прямо на ребрах
- stale deps видаляються після compute
- новий dependency-set формується в тому самому проході

Коли гілки постійно перемикаються, reflex не потрібно тримати
складну зовнішню структуру заради topo-repair або глобального рескану.

### 4. Batch updates

Сценарій:

- кілька writes підряд
- потім один етап читання або рендера

Чому reflex добра:

- `batchWrite()` рухає epoch один раз
- однаковий epoch дешево маркує пакет змін
- downstream лишається ліниво інвалідованим до читання

Це особливо корисно, коли запис іде пакетами, а споживання результату
відбувається пізніше.

### 5. Реалістичний partial UI

Сценарій:

- оновлюється багато даних
- читається лише видима частина екрана
- частина memo взагалі не зачіпається поточним рендером

Чому reflex добра:

- непрочитана частина графа лишається просто invalidated
- видима частина графа refresh-иться on demand
- лінивність перетворюється на дуже практичну економію роботи

Якщо у вас віртуалізований список, таблиця, viewport-based rendering
або часткове перемальовування, це природна зона сили reflex.

## Де reflex зазвичай програє

### High fan-in final read

Це головний слабкий кейс.

Сценарій:

- багато незалежних шляхів сходяться в один final memo
- після зміни все одно читається саме цей фінальний вузол

Наприклад:

```txt
many sources -> many path memos -> one final sum
```

Чому тут reflex може програвати eager runtime:

- підсумкове читання все одно змушує привести в актуальність усі потрібні шляхи
- lazy підхід уже не економить роботу, бо фінальний read вимагає все
- eager runtime міг зробити частину роботи заздалегідь на flush/update phase

Важливо:
це не баг reflex.
Це природна ціна lazy-дизайну.

## Що саме reflex намагається робити дешевим

### Дешевий write

При `signal.write(value)` runtime:

1. перевіряє `Object.is`
2. оновлює `payload`
3. ставить новий `t`
4. поширює `Invalid` вниз

І все.

Немає негайного recompute downstream-графа.

### Дешевий invalidation

`markInvalid()` не намагається:

- довести свіжість кожного вузла
- перерахувати memo
- перебудувати топологію

Він лише ставить propagation hint.
Це дуже важливо для профілю продуктивності.

### Локальний refresh замість глобального pass

Коли ви читаєте memo, reflex оновлює лише потрібний зріз графа.
Це дешевше, ніж "про всяк випадок" прогнати все.

## Як читати benchmark правильно

Бенчмарки в `signal_beta.bench.ts` корисні лише якщо розуміти,
що вони міряють різні режими роботи runtime.

Вони не відповідають на питання:

"Хто найкращий reactive runtime у світі?"

Вони відповідають на питання на кшталт:

- скільки коштує часткове читання після запису
- скільки коштує глибокий lazy-ланцюжок
- скільки коштує fan-in фінального значення
- що відбувається при dynamic deps
- як поводиться runtime при realistic UI workload

### Правильний спосіб інтерпретації

Якщо reflex швидша в якомусь сценарії, це зазвичай означає:

- лінивність зекономила реальну роботу

Якщо reflex повільніша, це зазвичай означає:

- фінальне читання все одно потребувало пройти майже все,
  і eager-pipeline мав вигідніший розподіл роботи

## Найважливіші cost-centers

Якщо дивитися на reflex як на машинку витрат, основні статті витрат такі.

### 1. Прохід по `firstOut` під час invalidation

Платимо за fan-out джерела.

Якщо один source має величезну кількість downstream-вузлів,
write-path усе одно повинен позначити їх invalid.

### 2. Прохід по `firstIn` під час freshness check

Платимо за кількість залежностей конкретного node.

Чим вищий fan-in, тим дорожча відповідь на питання:

"чи потрібен recompute?"

### 3. Реальний recompute функції

Це вже користувацька логіка.
Іноді саме вона домінує, а не сам runtime.

### 4. Tracking і cleanup динамічних ребер

Якщо dependency-set часто змінюється, runtime:

- позначає використані edges поточним `s`
- потім видаляє stale edges

Це нормальна й необхідна плата за коректні dynamic deps.

### 5. Effect scheduling

Для effect-вузлів є окрема вартість:

- enqueue
- dedupe
- flush/eager execution

У reflex scheduler зроблений мінімалістично, щоб цей шар не з'їдав виграш ядра.

## Чому `Object.is` такий важливий

Дуже часта помилка в perf-розмовах:

"ну перевірка рівності - дрібниця"

Ні, не дрібниця.

Якщо запис прийшов із тим самим значенням:

- `t` не повинен рухатися
- downstream не повинен invalidation-итися
- граф не повинен отримувати фальшиву роботу

No-op write - це не косметика, а основа реальної ефективності.

## Чому не можна сліпо робити eager flush всюди

Коли runtime бачить зміну, дуже хочеться "швидко все перерахувати".
Але це швидко лише в деяких графах.

У частково читаному UI eager-підхід часто робить зайву роботу:

- рендер прочитає лише частину даних
- а eager runtime уже перерахував значно більше

Тому в reflex scheduler для effects винесений окремо,
а lazy SAC-core залишений мінімальним.

## Практичні правила оптимізації reflex

### 1. Спочатку оптимізуйте модель читання, а не write-path

Якщо застосунок читає маленький зріз великого графа,
reflex вже добре розташована до виграшу.
Часто вигідніше покращити патерн читань, ніж переписувати ядро.

### 2. Не ламайте `t/v` семантику заради мікробенчмарку

Якщо заради красивої цифри почати:

- оновлювати `t` надто часто
- перестрибувати через cleanup
- трактувати `Invalid` як full-stale proof

ви майже напевно зламаєте коректність.

### 3. Вимірюйте build-up і steady-state окремо

У reactive runtime є різні фази:

- створення графа
- priming/first read
- write/read steady-state

Змішувати їх в одну цифру шкідливо.

### 4. Дивіться на патерн fan-in/fan-out

Два графи однакового розміру можуть поводитися дуже по-різному:

- один дешевий на invalidation і дорогий на final read
- інший навпаки

Форма графа важливіша за просте число вузлів.

### 5. Не забувайте про частоту branch switching

Якщо dependency-set змінюється постійно, вартість cleanup стає
реальною частиною профілю.
Це не погано, просто це треба враховувати.

## Типові perf-пастки під час розвитку ядра

### Пастка 1. Додати "зручну" абстракцію на hot path

Кожен зайвий універсальний шар у місцях на кшталт:

- `trackRead`
- `ensureFresh`
- `recompute`
- scheduler enqueue/flush

може з'їсти помітну частину профілю.

### Пастка 2. Використовувати важкі структури заради краси API

Наприклад:

- `Set` там, де вистачає marker-біта
- `Map` там, де можна пройти однозв'язний список
- окремі heap-об'єкти заради елегантної архітектури

У маленькому runtime це часто програш.

### Пастка 3. Тягнути dev-safety у prod

Додаткові asserts і докладні guards корисні в розробці,
але вони не повинні жити в production bundle.

### Пастка 4. Лікувати слабкий сценарій ціною втрати сильних

Якщо намагатися будь-якою ціною покращити high fan-in final read,
можна легко погіршити:

- partial reads
- cheap writes
- dynamic deps
- memory footprint ядра

Треба покращувати профіль адресно, а не розмивати дизайн.

## Коли reflex - дуже хороший вибір

Зазвичай reflex особливо доречна, якщо у вас:

- багато derived state
- часткові читання
- UI, який бачить лише частину даних
- lazy-friendly access pattern
- бажання тримати runtime маленьким і зрозумілим

## Коли варто бути обережнішим

Якщо ваш workload майже завжди означає:

- "після кожного запису прочитати підсумковий глобальний результат"
- "у графа дуже високий fan-in в одну фінальну точку"
- "лінивність майже нічого не економить"

тоді reflex може виявитися не найкращою точкою оптимуму.

## Короткий висновок

Продуктивність reflex будується не на магічному "швидше за всіх",
а на дуже конкретному профілі:

- дешевий запис
- дешева інвалідація
- селективна оплата під час читання
- сильна робота з partial read і dynamic deps
- природна слабкість у сценаріях, де фінальне читання все одно тягне весь граф

Якщо розуміти цей профіль, бенчмарки перестають бути суперечкою смаків
і стають нормальним інженерним інструментом.
